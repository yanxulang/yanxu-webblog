# 从零理解 yanxu-webblog

这个项目的重点不是博客功能本身，而是展示一个言序 Web 请求如何跨越 HTML、HTTP、框架和数据格式四层。你可以先运行现成项目，再按下面顺序阅读和改造。

## 1. 启动项目

```sh
git clone https://github.com/yanxulang/yanxu-webblog.git
yanbao --manifest-path yanxu-webblog install
yanbao --manifest-path yanxu-webblog run
```

另开终端验证所有响应：

```sh
curl -i http://127.0.0.1:8787/
curl -i http://127.0.0.1:8787/posts/hello-yanxu
curl -i http://127.0.0.1:8787/api/posts
curl -i http://127.0.0.1:8787/api/posts.yj
curl -i http://127.0.0.1:8787/static/app.css
curl -i http://127.0.0.1:8787/health
```

## 2. 数据层

`src/数据.yx`导出`全部文章`和`查文章`。每次调用都返回新容器，避免请求之间共享可变列。文章是包含`id`、`title`、`summary`、`body`和`date`的典。

先尝试添加第三篇文章。只修改数据层后，首页、详情、JSON API 和言据 API 都会同时出现新内容，这说明四种响应读取同一个领域数据源。

## 3. HTML 页面层

`src/页面.yx`只依赖`yanxu-html`，不关心 HTTP。它包含：

- `布局`：生成完整 HTML5 文档；
- `文章卡`：返回一个可复用节点；
- `首页`：用`HTML.组件`延迟构建文章列；
- `文章页`和`未找到页`：返回页面节点。

文章正文包含`<标签>`，但它通过`HTML.文字`进入页面，所以浏览器只会看到普通文字。页脚的`&copy;`是可信常量，使用`HTML.原始`显式绕过转义。搜索`HTML.原始（`即可审计项目内全部原始 HTML 信任点。

## 4. 应用层

`src/应用.yx`把页面和数据映射为响应：

```yanxu
所应用.取（「/」，首页处理）；
所应用.取（「/posts/:id」，文章处理）；
所应用.取（「/api/posts」，JSON接口）；
所应用.取（「/api/posts.yj」，言据接口）；
所应用.取（「/health」，健康检查）；
所应用.挂静态（「/static」，静态根）；
```

`文章处理`用`上下文.参数值（「id」）`取得路径参数。文章存在时返回安全 HTML 200，不存在时用`HTML状态响应`返回保持相同布局的 404。

JSON 由`Web.JSON响应`直接序列化；言据先调用`言据.序列化`，再交给`Web.言据响应`设置媒体类型。这保持了数据格式与 HTTP 传输的边界。

## 5. 横切能力

`安全首部`是一个后置响应中间件：它先调用`下一步（）`取得响应，再统一添加三个首部。因此正常页面、API、404、405 和静态文件共享同一策略。

`错误页`是应用的统一错误处理器。当前返回不含内部错误细节的 500 文字响应；实际项目可以同时写入受保护日志，但不要把堆栈或敏感值返回给客户端。

## 6. 服务器入口

`src/主.yx`只做三件事：

1. 从`YANXU_WEBBLOG_ADDR`读取可选监听地址；
2. 根据当前工作目录选择静态文件根；
3. 创建开发服务器，并按`--once`选择单请求或持续运行。

把地址配置放在入口而不是应用模块，使`tests/集成.yx`可以直接构造应用，不打开端口。

## 7. 集成测试

测试通过`HTTP.解析请求头`创建真实`HTTP请求`，直接调用`应用.处理`，再检查状态码、首部和正文字节。这样覆盖完整路由/中间件/页面链，同时保持快速、确定且不占用 TCP 端口。

```sh
yanxu 试 yanxu-webblog/tests --json
```

建议扩展练习：新增`POST /api/posts`并验证 405 变化、添加查询过滤、为 JSON 404 建立独立错误结构，或把`数据.yx`替换为文件存储。当前 HTTP 0.1 尚未提供 Multipart 和流式请求，扩展时要遵守[部署边界](running-and-deployment.md)。
