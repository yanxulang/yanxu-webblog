# yanxu-webblog 项目结构

项目按“数据 → 页面 → 应用 → 进程入口”保持单向依赖，让每层可以独立测试和替换。

## 文件职责

| 路径 | 职责 | 主要依赖 |
| --- | --- | --- |
| `src/数据.yx` | 返回和查询文章领域数据。 | 无第三方包 |
| `src/页面.yx` | 把领域数据转换为 HTML 节点。 | `yanxu-html` |
| `src/应用.yx` | 处理器、API、路由、中间件、错误边界。 | `yanxu-web`、`yanxu-http`、`yanju` |
| `src/主.yx` | 环境配置和服务器生命周期。 | `yanxu-web` |
| `static/app.css` | 静态资产示例。 | 静态文件服务 |
| `tests/集成.yx` | 直接驱动应用的跨库集成测试。 | 全部四个包 |

## 依赖固定方式

`言序.toml`声明四个 Git 来源与 SemVer 要求，`言序.lock`记录每个包的精确提交、版本、入口和内容 SHA-256。言包把源码放入统一缓存；仓库内没有`deps/`副本或 submodule。

```text
言序.toml  ──来源与版本要求──>  yanbao install / update
                                      │
                                      ▼
言序.lock  <──精确提交与校验和──  Git 包缓存
```

日常克隆后运行`yanbao --manifest-path yanxu-webblog install`，它优先验证现有锁定。只有准备升级时才执行`yanbao update`；更新后应审查锁文件差异并运行完整测试，不要手工改写提交或校验和。

## 常见扩展位置

- 新页面组件：放入`页面.yx`，或在增长后拆为`src/页面/`模块；
- 新 API：在`应用.yx`添加处理器和路由，复杂后拆为独立接口模块；
- 持久数据：以新的数据模块替换`数据.yx`，不要让页面直接访问文件或数据库；
- 身份验证：实现中间件并把已验证身份放入请求可访问的应用约定中；
- 配置：在`主.yx`读取清单允许的环境变量，再把显式配置交给应用工厂；
- 静态构建：页面模块只返回 HTML 节点，因此可在独立工具中直接渲染，不必启动 HTTP。

## 测试策略

优先通过`创建应用`进行无网络集成测试。对数据查询和页面节点增加独立测试后，再用少量`--once`冒烟测试覆盖真实 TCP 连接。这样能把协议、框架和业务失败定位到最小层。
